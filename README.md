# Proyecto Web Colaborativo

## Objetivo del proyecto

Construir una pagina web sencilla usando HTML y CSS para practicar un flujo de trabajo colaborativo con Git y GitHub. El proyecto demuestra el uso de una rama principal, ramas de desarrollo, commits descriptivos, pull requests, revision de codigo y resolucion de conflictos.

## Integrantes y roles

| Integrante | Rol | Responsabilidad |
| --- | --- | --- |
| Cesar | Lider | Coordinar tareas, validar avances y revisar que se cumplan los requisitos. |
| Integrante 2 | Documentador | Mantener el README y registrar el flujo de trabajo usado. |
| Integrante 3 | Integrador | Gestionar ramas, pull requests, revisiones y fusiones. |
| Integrante 4 | Disenador | Definir estilos, colores, distribucion visual y experiencia de usuario. |

> Reemplazar los nombres de ejemplo por los integrantes reales del equipo.

## Flujo de trabajo usado

1. Se inicializo el repositorio local con `git init`.
2. Se uso `main` como rama principal estable.
3. Se creo la rama `dev` para integrar avances generales del proyecto.
4. Se creo la rama `diseno` para trabajar ajustes visuales y de estilos.
5. Cada integrante realizo commits con mensajes claros y descriptivos.
6. Los cambios se subieron a GitHub y se abrio un pull request hacia `main`.
7. Otro integrante reviso el pull request y dejo comentarios de revision.
8. Se resolvieron conflictos cuando dos ramas modificaron la misma seccion del proyecto.
9. Una vez revisado y corregido, el integrador fusiono los cambios en `main`.

## Ramas requeridas

- `main`: version principal del proyecto.
- `dev`: rama de desarrollo general.
- `diseno`: rama para cambios visuales.

## Revision de codigo

El pull request debe ser revisado por un integrante distinto al autor del cambio. La revision debe confirmar que el sitio abre correctamente, que el README explica el objetivo del proyecto y que los mensajes de commit son claros.

## Ejemplos de commits descriptivos

- `Crear estructura inicial del sitio web`
- `Agregar estilos principales de la pagina`
- `Documentar objetivo roles y flujo de trabajo`
- `Resolver conflicto en seccion de flujo de trabajo`

## Como ver el proyecto

Abrir el archivo `index.html` en un navegador web.

## Evidencia sugerida para entrega

- Captura del repositorio en GitHub.
- Captura de las ramas `main`, `dev` y `diseno`.
- Captura del pull request con revision de otro integrante.
- Captura del conflicto resuelto o del commit que registra la resolucion.
